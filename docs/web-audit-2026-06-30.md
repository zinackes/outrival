# Web App — UX / UI & Design-System Findings

Consolidated from cross-surface input (landing, auth/onboarding, shell/nav, signals/competitors, secondary views, settings, design-system primitives). No findings are graded Critical, so that group is omitted. All input is unverified; items resting on medium-confidence observations are tagged **⚠ verify**.

### Systemic patterns (recur across many components)
- **Uppercase letterspaced "micro-eyebrow" casing** — banned by house rules, yet baked into a shared primitive (`dropdown-menu.tsx:145`) so every menu inherits it, plus nav/settings/billing/onboarding/dead components. → Medium item below.
- **Unconverted shadcn primitives** — Textarea/Select/Toggle/Accordion/Tabs/Command still ship stock defaults while Input/Checkbox/Button were migrated, diverging on surface, focus ring, disabled fade, height. → High + Medium + Low items below.
- **Hardcoded colors / token bypass** — raw hex, `bg-white/N`, `text-white`, and `*-[var(--token)]` arbitrary syntax instead of mapped utilities. → Medium + Low items below.
- **Geist Mono on non-data prose** — mono leaking onto labels, relative times, "via Exa.ai". → Low item below.
- **Loading / empty / error states** — bare `<p>Loading…</p>`, spinners, and hand-rolled panels instead of the shared `Skeleton`/`EmptyState`/`ListError` system; one case loops forever. → Medium + Low items below.
- **Redundant uppercase plan eyebrow printing the plan name twice** — same component shape on landing pricing and in-app billing. → Medium item below.

## High

- **Landing primary CTAs scroll to `#cta` instead of starting signup, and plan-specific pricing buttons discard the chosen plan.** `hero.tsx:53`, `nav.tsx:66`, `sources.tsx:127`, `pricing.tsx:147-153`, `cta.tsx:24`. Identical "Start free" labels behave two ways (anchor-scroll vs the one real `/auth` link), and every paid pricing CTA points at `#cta`, throwing away the selected plan. Fix: make every start/try CTA a `Link` to `/auth`, pass the plan from pricing cards (`/auth?plan=pro`), and reserve `#cta` only for genuine "jump to" links.

- **OnboardingBanner warning uses the neutral `accent` token → invisible gray-on-gray.** `onboarding-banner.tsx:18-19`. `border-accent/30 bg-accent/10` + `text-accent` icon; but `--color-accent` maps to `--surface-3` (neutral elevated gray, `globals.css:247`), so in dark mode the warning icon and CTA render ~#242424 on ~#161616. Fix: use a semantic token — `border-destructive/30 bg-destructive/10` + `text-destructive` for a warning, or `border-primary/30 bg-primary/10 text-primary` for a brand nudge. Never use Tailwind `accent` as a foreground/fill.

- **Ask FAB and Feedback FAB occupy the exact same fixed slot — one is unclickable.** `ask-dock.tsx:49`, `feedback-widget.tsx:96` (both `fixed bottom-5 right-5 z-40`); consent banner crowds the same corner (`consent-banner.tsx:33`). Whichever paints last covers the other. Ask already has a topbar button + sidebar item. Fix: give the corner one owner — offset the FABs (e.g. Feedback `bottom-5`, Ask `bottom-20`) or drop the redundant Ask FAB; they must not share bottom/right/z.

- **Form-field primitives split into two token dialects, and `dark:bg-input/30` paints the border token as a field surface.** `textarea.tsx:10`, `select.tsx:40`, `toggle.tsx:10,16`, `accordion.tsx:38` (vs converged `input.tsx:14-15`, `checkbox.tsx:17`, `button.tsx:12`); token map `globals.css:254`. The unconverted controls keep `bg-transparent` + `dark:bg-input/30`, `shadow-xs`, `focus-visible:ring-ring/50`, `disabled:opacity-50`, and a 36px height vs Input's 38px. Worse, `--color-input` = `var(--border)`, so `dark:bg-input/30` resolves to a faint translucent-white wash, not the deep `--field` surface — a textarea/select beside an input reads as a milky grey panel. Fix: converge onto the Input recipe — `bg-field`, drop `shadow-xs` and all `dark:bg-input/*`, align height to `h-[38px]`. This is the single biggest primitive-layer inconsistency.

`apps/web/src/app/dashboard/layout.tsx:86-137`; `apps/web/src/components/dashboard/sidebar-competitors.tsx:50`; `apps/web/src/components/dashboard/sidebar.tsx:146`; `apps/web/src/components/outrival/ai-status-banner.tsx:21`; `apps/web/src/components/outrival/structural-change-banner.tsx:22`
The layout already does 4 parallel server fetches and the SSR-seed machinery exists, but the chrome mounted on every route (competitor roster, product switcher, AI-status + structural-change banners) is seeded on only 2 of ~14 routes. Hard-loading `/dashboard/signals|activity|trends|discovery|settings/*` fires 4 fresh client round-trips after hydration.
Fix: in `layout.tsx` build a per-request `makeServerQueryClient()`, prefetch the chrome queries with the same `h` headers, and wrap the shell in `<HydrationBoundary state={dehydrate(qc)}>` — the pattern the pages already use.

**Two always-on 60s background polls in the chrome; the sidebar poll refetches the full competitor roster (with per-competitor stats) just to refresh a count badge.**
`apps/web/src/components/dashboard/sidebar-competitors.tsx:24,50`; `apps/web/src/components/outrival/ai-status-banner.tsx:8,24`
`refetchInterval: 60_000` on `competitorsQuery` re-pulls the heavy roster (signals7d, lastSignalAt) every minute for the whole session; `AiStatusBanner` independently polls every 60s. The detail view already invalidates `competitorsQuery` on mutations (`competitor-detail-view.tsx:505/524/571`), so the roster stays fresh without a fixed poll.
Fix: drop the sidebar `refetchInterval` (rely on invalidation + window-focus refetch) or widen it substantially; gate the AI-status poll on `document.visibilityState` — rate-limit incidents are not sub-minute.

**Compare matrix and sector feed fetch via raw `useEffect`, bypassing TanStack Query (no cache, dedup, or hydration).**
`apps/web/src/components/dashboard/compare-view.tsx:627-731`; `apps/web/src/components/dashboard/sectoral-feed.tsx:35-95`; `apps/web/src/app/dashboard/sector/page.tsx:1-14`
Both hand-roll `matrix`/`isFetching`/`cancelled` state. Re-selecting a previously-compared set or toggling a sector category back re-hits the network from scratch; Sector also passes `initialSignals` via props instead of the dehydrate/HydrationBoundary contract every other page uses.
Fix: move the matrix to `useQuery({ queryKey:['compare', [...selected].sort()], placeholderData: keepPreviousData })`; add a `sectoralFeedQuery({category,view,offset})` factory, seed `sector/page.tsx` via `setQueryData` + HydrationBoundary, and use `useInfiniteQuery` for Load-more.

**Trends SSR prefetch is silently wasted by a timezone-dependent date query key.**
`apps/web/src/app/dashboard/trends/page.tsx:18-22`; `apps/web/src/lib/queries.ts:96-106`; `apps/web/src/components/dashboard/trends-view.tsx:170,177`
The key embeds `range.from/to.toISOString()`; the server computes `endOfDay(new Date())` in UTC while the client recomputes it in local time → different ISO instants → the dehydrated entry is stored under a key the client never requests, so `useQuery` misses and refetches.
Fix: key on a date-only bound (`format(from,'yyyy-MM-dd')`) or derive the 90d window from UTC day boundaries on both sides so server seed and first client render produce identical keys.

**Overview and Signals page hold two cache entries for the same 200-signal set (keys differ only by server sort).**
`apps/web/src/app/dashboard/page.tsx:22`; `apps/web/src/components/dashboard/overview.tsx:126`; `apps/web/src/app/dashboard/signals/page.tsx:21`; `apps/web/src/components/dashboard/signals-view.tsx:145`
Overview reads `['signals',{limit:200}]`; Signals reads `['signals',{limit:200,sort:'threat'}]`. Navigating between them refetches all 200 despite the 60s staleTime.
Fix: have Overview read the same key (`sort:'threat'`); it re-sorts client-side anyway (`SEV_ORDER`), so one shared entry serves both.

**Competitor-detail scrape poller double-fetches the detail and broad-invalidates every tab query each tick.**
`apps/web/src/app/dashboard/competitors/[id]/competitor-detail-view.tsx:331-334,381-453`
The poll calls `refresh()` (refetches `['competitor',id,'detail']`) then `invalidateQueries({ queryKey:['competitor',id] })` — the prefix re-fetches the detail again plus every mounted tab query (`jobs`/`pricingHistory`/…) regardless of staleTime.
Fix: drop the redundant detail invalidate (`refresh()` already updated the cache), or scope the invalidate to the specific finished source's tab key.

**Master-list rows are unmemoized and format dates inline → the whole list re-renders on every selection/keystroke (systemic across 3 lists).**
`apps/web/src/components/dashboard/signal-row.tsx:42-114,121-192,103,184`; `apps/web/src/components/dashboard/signals-view.tsx:788-806`; repeats in `apps/web/src/components/dashboard/overview.tsx:711-799,787` and `apps/web/src/components/dashboard/competitors-list.tsx:547-664,627`
`SignalRow`/`BatchRow` are plain functions calling `formatDistanceToNow(new Date(...))` in render. A `focusedId` change re-renders all up-to-200 rows (each re-parsing a Date) when only 2 rows' `selected` flipped.
Fix: wrap rows in `React.memo`, precompute the relative-time string per row, and consider `@tanstack/react-virtual` for the 200-row lists.

**Editable forms and sheets fork server state into a local draft and never write back, leaving siblings stale (systemic, 6+ components).**
`apps/web/src/components/outrival/notification-settings-form.tsx:41-78`; `apps/web/src/components/outrival/notification-moderation-form.tsx:81-124`; `apps/web/src/components/outrival/workspace-settings-form.tsx:58-141`; `apps/web/src/components/dashboard/digest-settings-sheet.tsx:37-45`; `apps/web/src/components/dashboard/alert-channels-sheet.tsx:60-68`; `apps/web/src/components/outrival/detection-config-sheet.tsx:51-62`
Each PATCHes then updates only its local `pristine` state, never `queryClient.setQueryData` — so `["notificationSettings"]` (shared by IntegrationsSettings, AlertChannelsSheet, DigestSettingsSheet) and `["workspaceSettings"]` (DeleteWorkspaceCard's confirm name) serve the old value for up to 60s (`query-provider.tsx:19`). The same `useQuery → useState(draft)+useRef(initialized)+sync-effect` boilerplate is re-implemented in all of them, and the sheets' `effect` keyed on `settingsQ.data` can clobber in-progress edits on a background refetch.
Fix: write through on success (`setQueryData(notificationSettingsQuery().queryKey, updated)` / invalidate) and extract a shared `useEditableQuery(queryOptions)` returning `{draft,pristine,dirty,reset,commit}`; at minimum guard the sheet sync-effects to seed once per open (when `settings===null`).

**2FA toggle force-remounts the whole `SignInMethods` subtree, re-fetching accounts and passkeys.**
`apps/web/src/components/outrival/security-settings.tsx:1167,574-576`
`<SignInMethods key={String(twoFactorEnabled)} … />` uses `key` as a remount hammer because `TwoFactorRow` stores `useState(initialEnabled)` instead of deriving from the prop; toggling 2FA tears down/re-fetches `ConnectedAccountRows` and `PasskeysRow`.
Fix: drive `TwoFactorRow` from the prop (or lift state) and remove the `key`.

**Heavy client-only libraries are statically imported into route bundles instead of code-split (systemic).**
`apps/web/src/components/ui/date-range-picker.tsx:10,156` → `react-day-picker` ships in every page with a date filter (Overview, Trends, Digests), though Calendar only renders inside the "Custom range" popover; `apps/web/src/components/outrival/security-settings.tsx:7,330-332` → `qrcode.react` ships on every Security visit for a once-ever 2FA enrollment; `apps/web/src/components/dashboard/competitors-list.tsx:7,546,673-675` → `motion/react` pulled in just for row enter/exit; `apps/web/src/app/dashboard/competitors/[id]/competitor-detail-view.tsx:135-141` → all eight tab modules bundled though one renders at a time.
Highest leverage is `react-day-picker` (primary dashboard surfaces). Fix: `next/dynamic(() => import("@/components/ui/calendar").then(m => m.Calendar), { ssr:false })`; dynamic-import the 2FA dialog and non-default competitor tabs; replace `motion/react` rows with a CSS `animate-in fade-in`.

**Static content shipped as `'use client'` components ('use client' at the wrong altitude) (systemic).**
`apps/web/src/components/landing/digest-mockup.tsx:1,87-106` (only call site passes `animate={false}`, so both effects early-return — the largest landing block hydrates for zero interactivity); `apps/web/src/components/dashboard/whats-new-view.tsx:1-104` (whole changelog hydrated just to run one `localStorage.setItem`); `apps/web/src/components/ui/table.tsx:1` (purely presentational, forces any server parent into client rendering — *low-confidence*)
Fix: render the markup in a server component and extract only the imperative bit into a tiny client child (`<MarkWhatsNewSeen />`); drop `DigestMockup`'s `animate` prop on the landing path; delete `"use client"` from `table.tsx`.

**`mode` is effect-synced state that should be derived from session.**
`apps/web/src/app/(onboarding)/onboarding/onboarding-form.tsx:273-280`
`useState` + a `modeAdopted` ref + an effect copy `session.mode` in once; `setMode` is never called elsewhere — the triple adds a dead extra render.

**Onboarding session is loaded client-side after hydration when the RSC page could seed it (low-confidence).**
`apps/web/src/hooks/use-onboarding-session.ts:16-40`; `apps/web/src/app/(onboarding)/onboarding/page.tsx:44-66`

**`DateRangePicker` recomputes preset ranges (~18 Date allocations) several times per render and has an exhaustive-deps violation.**

**`transition-all` on interactive primitives instead of scoped transitions (systemic; low-confidence).**
`apps/web/src/components/ui/button.tsx:12`; `apps/web/src/components/ui/progress.tsx:24`; `apps/web/src/components/ui/accordion.tsx:38`

**Two component-authoring conventions mixed across `ui/*` (forwardRef vs data-slot).**
`apps/web/src/components/ui/command.tsx:10`, `dropdown-menu.tsx:16`, `dialog.tsx:14`, `popover.tsx:12` (legacy `forwardRef`+displayName) vs `button.tsx:56`, `select.tsx:9`, `tabs.tsx:9`, `sheet.tsx:9`, `table.tsx:7` (function + `data-slot`)
Copy-paste drift across two shadcn generations → inconsistent ref behavior and non-uniform `[data-slot=...]` styling hooks.
Fix: standardize on the React 19 `data-slot` function-component form; ensure every primitive both forwards ref and emits a `data-slot`.

**`ToggleGroup` has a dead `data-[spacing=default]` selector and an opaque unitless `--gap` var.**
`apps/web/src/components/ui/toggle-group.tsx:37,39`
`spacing` is a number, so `data-spacing` is never `"default"` — `data-[spacing=default]:...shadow-xs` can never match (the working branch keys off `data-[spacing=0]`).
Fix: remove the unreachable class or restore a real `default` sentinel and key all selectors off it.

**Shared `feedItemMotion` defaults to FLIP `layout: true` + spring on feeds that are not virtualized (low-confidence).**
`apps/web/src/lib/motion.ts:15-30`; consumers `signals-view.tsx`, `competitors-list.tsx`, `discovery/discovery-view.tsx`
